# ExecuTorch CMSIS-Pack — all-ops coverage test

A self-contained [CMSIS-Toolbox](https://github.com/Open-CMSIS-Pack/cmsis-toolbox)
solution that exercises the **entire operator surface** of the ExecuTorch
CMSIS-Pack on a Cortex-M55 target (Arm Corstone-300 / `ARMCM55`).

It gives every operator the pack ships two kinds of coverage:

- **Build / link** — [all_ops.cproject.yml](all_ops.cproject.yml) selects every
  ExecuTorch operator component, so linking the firmware exercises each op's
  generated registration, forward declaration and kernel source.
- **Execution** — 165 per-operator test models (`model.pte` + reference
  `input_*.bin` / `expected_*.bin`) are embedded directly into the ELF `.rodata`
  via [embedded_models.S](embedded_models.S). The firmware runs every model on
  the ExecuTorch runtime, compares against the expected output, and prints
  `PASS`/`FAIL` per op plus a final aggregate — no semihosting filesystem access
  needed at run time.

Models covered: **149 Portable + 16 Cortex-M = 165** (see
[SKIPPED_OPS.md](SKIPPED_OPS.md) for ops that are build/link-covered but not
executed, and why).

## Prerequisites

- [CMSIS-Toolbox](https://github.com/Open-CMSIS-Pack/cmsis-toolbox) ≥ 2.13
  (`cbuild`, `csolution`, `cpackget`)
- `arm-none-eabi-gcc` 14.x
- The **ExecuTorch CMSIS-Pack** (`PyTorch::ExecuTorch`) installed, plus
  `ARM::CMSIS`, `ARM::Cortex_DFP` and `ARM::CMSIS-NN`:

  ```bash
  cpackget add PyTorch.ExecuTorch.<version>.pack
  ```

## Build

```bash
cbuild all_ops.csolution.yml
```

The image is written to `out/all_ops/ARMCM55/Debug/all_ops.elf`.

## Run

On the **Arm FVP** (Corstone-300):

```bash
FVP_Corstone_SSE-300 \
  -C mps3_board.uart0.out_file=- \
  -C mps3_board.uart0.shutdown_on_eot=1 \
  -C cpu0.semihosting-enable=1 \
  out/all_ops/ARMCM55/Debug/all_ops.elf
```

On **hardware** via a CMSIS-DAP probe (pyOCD): use the `CMSIS Load+Run` task in
[.vscode/tasks.json](.vscode/tasks.json), or `pyocd load` / `pyocd gdbserver`
with the generated `*.cbuild-run.yml`.

`.vscode/` ships ready-made FVP (`fvp-gdb`) and pyOCD debug tasks/launch
configs.

## Regenerating the embedded models

The committed `embedded_models.{S,cpp,h}` are produced from `models/` by:

```bash
python3 gen_embedded_models.py
```

Re-run it whenever `models/manifest.json` or any model directory changes. The
`.incbin` paths are emitted **relative** to the project directory and resolved
via the cproject's `add-path: .`, so the project relocates cleanly.

## Layout

| Path | Purpose |
|------|---------|
| `all_ops.csolution.yml` / `all_ops.cproject.yml` | CMSIS solution selecting every operator component |
| `main.cpp` | All-ops runner: loads, runs and checks each embedded model |
| `models/` | Per-op `.pte` + reference input/expected vectors (`manifest.json` indexes them) |
| `embedded_models.{S,cpp,h}` | Models embedded into `.rodata`; generated by `gen_embedded_models.py` |
| `et_pal_override.cpp` / `retarget_semihosting.cpp` | Route `ET_LOG()` / stdout to semihosting |
| `ARMCM55_large.ld` | Linker script sized for the embedded model blob |
